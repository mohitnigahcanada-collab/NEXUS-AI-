# CORE IDEA

BuilderBrain is a local AI Engineering Brain.

Before any AI coding agent starts coding, debugging, planning, or refactoring, it should first:

1. Search local knowledge.
2. Understand user style.
3. Select the right stack of books.
4. Generate a combined context pack.
5. Create a safe execution proposal with confidence/risk.
6. Ask approval for risky actions.
7. Execute only after context.
8. Verify work.
9. Save lessons for next time.

Bad workflow:

Task -> AI guesses -> AI codes -> bugs happen

Good workflow:

Task -> BuilderBrain detects domain -> selects knowledge books -> builds context pack -> creates proposal -> shows confidence/risk -> executes safely -> verifies -> saves lessons

Later this system can become a big MCP-connected knowledge brain for Claude Code, OpenCode, Codex, Antigravity, ChatGPT, OpenHands, local models, and other AI agents.

But for v0.1, build the basic verified core first.

# COMMAND: brain context

Example:

brain context "fix login redirect bug"

It should:

1. Read pocket rules.
2. Read safe user style.
3. Classify the task into domains.
4. Select multiple relevant books, not just one.
5. Build one combined context pack.
6. Include relevant rules.
7. Include best patterns.
8. Include anti-patterns.
9. Include previous lessons if relevant.
10. Include testing checklist.
11. Include recommended next plan.
12. Save run log.

Output structure:

TASK:
<task>

DETECTED DOMAINS:
- auth
- routing
- debugging
- frontend
- testing

BOOK STACK USED:
- pocket-rules/before-coding.md
- pocket-rules/before-debugging.md
- mini-book/debugging.md
- mini-book/software-engineering.md
- mini-book/testing.md
- mini-book/security.md
- user-style/do-not-ask-rules.md
- self-learning/bug-patterns.md

CONTEXT PACK:
- key rules
- relevant patterns
- anti-patterns
- known previous lessons
- recommended approach
- testing checklist

RECOMMENDED NEXT ACTION:
Create a small plan, inspect relevant files, make reversible changes, test the flow.

# COMMAND: brain propose

Example:

brain propose "change auth routing to fix redirect loop"

It should produce an action proposal.

Proposal must include:

- task summary
- detected domains
- confidence level
- risk level
- approval required: yes/no
- why approval is required or not
- evidence used
- planned actions
- files likely to inspect/change
- rollback plan
- testing plan
- final recommendation

Approval is required for risky tasks involving:

- auth
- payments
- database migrations
- deletion
- deployment
- secrets
- security
- running unknown code
- changing production config

Example output:

ACTION PROPOSAL

Task:
Change auth routing to fix redirect loop.

Confidence:
Medium / 72%

Risk:
High

Approval Required:
Yes

Reason:
Touches auth and protected route behavior.

Evidence:
- before-debugging.md
- debugging.md
- security.md
- previous bug-patterns.md

Plan:
1. Inspect current auth flow.
2. Find duplicate redirects.
3. Add session hydration guard.
4. Make smallest reversible fix.
5. Test login/logout/OAuth flow.

Rollback:
Create backup branch or commit before modifying files.

Testing:
- email login
- OAuth login
- logout
- protected page refresh
- invalid session

# PROPOSAL ENGINE

The proposal engine should calculate:

Risk level:

- Low
- Medium
- High
- Critical

Confidence level:

- Low
- Medium
- High
- numeric percentage if easy

Simple scoring is fine.

Confidence increases when:

- domains detected clearly
- multiple books found
- previous lessons exist
- task is specific

Confidence decreases when:

- task is vague
- risky domains involved
- no relevant books found

Risk increases when task involves:

- auth
- payments
- database
- security
- deployment
- deletion
- secrets
- production

Approval required if risk is High or Critical.

# POCKET RULES CONTENT

Seed pocket rules with strong content.

before-coding.md:

- Do not code before understanding the task.
- Retrieve context first.
- Prefer small reversible changes.
- Do not guess when knowledge exists.
- Create a plan before edits.
- Verify before claiming done.

before-debugging.md:

- Reproduce or understand the failure.
- Identify root cause.
- Do not keep guessing after two failed attempts.
- Search previous bug patterns.
- Make the smallest fix.
- Test the exact failing flow.

approval-rules.md:

Approval required for:

- auth
- payments
- database migrations
- production deployment
- deleting data/files
- secrets/API keys
- security permissions
- running unknown repo code

memory-rules.md:

- Save solved lessons.
- Save failed attempts.
- Save reusable fixes.
- Update user style only with safe preferences.
- Do not store sensitive personal data unless explicitly asked.
